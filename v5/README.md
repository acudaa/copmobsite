# Copernicus for Transport & Mobility

A EUSPA-oriented static website showing how Copernicus Earth observation supports five transport and mobility segments, with use cases, case studies and news. It runs on plain static hosting (e.g. GitHub Pages).

## Two layers: engine vs content
The site is split so **content and structure evolve independently**:

- **Engine (structure)** — the HTML pages, `app.js`, `style.css`. Edit these to change design or behaviour. They never need rewriting when content grows.
- **Content (data)** — everything under `content/`. You add/grow this yourself by dropping JSON files. The engine reads it at load time.

The contract between them is the schema (see `content/SCHEMA.md`).

## How content loads
On load the site reads `content/manifest.json`, then fetches each file it lists and assembles the site. The manifest is an **index of which files exist** — it is generated by a script, never hand-edited.

```
content/
  site.json            site title/tagline (singleton)
  services.json        Copernicus services (singleton)
  segments.json        the 5 segments (singleton)
  usecases/    uc-*.json     one file per use case
  casestudies/ cs-*.json     one file per case study
  news/        *.json         one file per news item
  manifest.json        GENERATED by generate_manifest.py
  _templates/          copy these to make new items (ignored by the indexer)
  SCHEMA.md            field reference + how-to
```

## Adding / editing content (you, anytime)
1. Create or edit a file under `content/usecases/`, `content/casestudies/` or `content/news/` (copy from `content/_templates/`). Filenames can be anything ending in `.json`; the `id` inside is what counts. Files starting with `_` are ignored.
2. Run the indexer from the repo root:
   ```bash
   python generate_manifest.py
   ```
3. Commit. The item appears on the site.

To remove an item, delete its file and rerun the script. Segments and services are single files (`content/segments.json`, `content/services.json`) — edit them directly and rerun the script (it re-validates references).

The indexer **refuses to write the manifest** (exit code 1) on invalid JSON, a missing required field, or a duplicate `id`, and prints non-blocking warnings for unknown segment ids or dangling `relatedCaseStudies`. So you can add content confidently — mistakes fail loudly at index time, not silently on the site.

### Optional: authoring panel
`admin.html` is a convenience for editing. It loads the live content and lets you download each item as its own `<id>.json` file (plus the singletons), telling you exactly which folder to drop it in. After placing files, run `generate_manifest.py`. The panel never touches the engine or the manifest.

## Pages (engine)
| File | Purpose |
|------|---------|
| `index.html` | Landing — Copernicus overview + segment entry points |
| `segments.html` | Segments overview; per-segment page (challenges + needs → use cases → case studies, role of Copernicus). Nav has a Segments dropdown of names. |
| `usecases.html` / `usecase.html` | Use case list (maturity-marked boxes, segment filter) / detail (maturity at top, role of Copernicus, related case studies) |
| `casestudies.html` / `casestudy.html` | Case study list / detail (maturity badge, EU-project flag) |
| `news.html` | News feed with segment filter |
| `about.html` | Purpose, use-case vs case-study distinction, maturity key, EUSPA's role |
| `admin.html` | Content authoring tool (per-file export) |
| `app.js` / `style.css` | Shared engine: nav/footer/filters/loader, styles |
| `generate_manifest.py` | Indexes `content/` into `content/manifest.json` |

## Deploying to GitHub Pages
1. Commit everything (including `content/manifest.json`).
2. **Settings → Pages → Deploy from a branch → `main` / `root`**.
3. Live at `https://USER.github.io/REPO/`.

Whenever you change content, run `python generate_manifest.py` before committing so the manifest matches the files.

> Tip: you can make the manifest auto-regenerate on every push with a small GitHub Action that runs `python generate_manifest.py` and commits the result — ask if you want that added later. For now it's a manual one-line step.

## Local preview
The site fetches JSON, so it needs a server (not `file://`):
```bash
python -m http.server 8000   # open http://localhost:8000
```

## Maturity levels
Both use cases and case studies carry one of **Research** (early-stage / proof of concept), **Trial** (piloted/demonstrated with users), **Operational** (routine use).

## Real EU projects
Case studies flagged `"real": true` reflect documented EUSPA / Horizon Europe projects — **SPATRA** (rail temperature/buckling), **BUILDSPACE** (urban digital twins), **Space4Cities** (urban public space). Others are illustrative; their figures are indicative.
