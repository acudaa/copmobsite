# Copernicus for Transport & Mobility

A EUSPA-oriented static website showing how Copernicus Earth observation supports five transport and mobility segments, through user needs, use cases, case studies, tools, products, and news. It runs on plain static hosting (e.g. GitHub Pages).

## Two layers: engine vs content
The site is split so **content and structure evolve independently**:

- **Engine (structure)** — the HTML pages, `app.js`, `style.css`. Edit these to change design or behaviour. They never need rewriting when content grows.
- **Content (data)** — everything under `content/`. You add/grow this yourself by dropping JSON files. The engine reads it at load time.

The contract between them is the schema set in `schemas/`. **Needs, use cases, case studies, and tools are one interlinked graph**, not independently-shaped collections. News sits alongside it, using the same shared building blocks for consistency, but deliberately outside the graph.

## The content model: a demand → pattern → evidence graph

```
                 addresses                     instantiates
   need / requirement  ───────────▶  use case  ◀───────────────  case study
        (why)                     (pattern, hub)                  (evidence)
                                                                       │
                                                              same schema as ──▶  tool
                                                            (folder, not field, distinguishes them)

   classification facets ──▶  content/products.json , content/market-segments.json
```

- **need** — a user/stakeholder need, the *why*. Lives in `content/needs/`.
- **use case** — a reusable EO application pattern that *addresses* a need: the hub of the graph. Lives in `content/usecases/`.
- **case study** — a real or illustrative deployment that *instantiates* a use case: concrete evidence. Lives in `content/casestudies/`.
- **tool** — an interactive demonstrator or application. Lives in `content/tools/`. **A tool has no schema of its own** — it validates against `case-study.schema.json` verbatim, because a tool is presented the same editorial way as a case study (narrative, classification, maturity). The only thing distinguishing a tool from a case study is which folder it's in; there's no `kind` field doing that work. A tool's launch/demo link reuses the same field a case study uses for its external link: `classification.site.project_link`.

These are connected only through **typed `relations`** (`{type, target, target_kind}`), not by ad hoc id arrays. The two edges this site relies on today:
- `use_case.relations: [{type:"addresses", target:"<need-id>", target_kind:"need"}]`
- `case_study.relations: [{type:"instantiates", target:"<usecase-id>", target_kind:"use_case"}]`

A tool could equally carry a `relations` entry pointing at a use case it demonstrates (`target_kind: "tool"` is a valid relation target throughout the schema set), though none currently do.

`app.js`'s `loadContent()` resolves these into convenient reverse-edge arrays at load time (`need._usecases`, `usecase._caseStudies`, etc.) so the page-rendering code reads simple arrays — only the content files themselves carry the graph structure.

**Faceting** (which segment, which Copernicus product, which EO technique) goes through a shared `classification` block on every content item — needs, use cases, case studies, tools, *and* news — referencing the two vocabularies:
- `content/products.json` — the EO product **taxonomy** (component → service/mission → product → sub-product), e.g. `clms` → `egms` → `egms-ortho`. `content/products.index.json` is a precomputed ancestor/descendant index (regenerate with `python product_taxonomy.py --write` if you edit `products.json`). Browsable on the **Products** page.
- `content/market-segments.json` — the 5 themed transport segments (Aviation, Rail, Road, Maritime, Urban) plus a couple of plain vocabulary entries (`transport`, `infrastructure`). Segments with a `site` block are the ones with their own themed landing page; the others are facets only. News additionally accepts the sentinel value `"all"` in `classification.market_segments` for items that aren't segment-specific.

**Site-specific fields that don't belong in the shared schema** (an integration-mode pill on use cases, an "is this a real EU project" flag + link on case studies/tools, per-segment Copernicus-role prose) live under `classification.site` — a deliberately open extension point, not part of the cross-content model. See `schemas/common.schema.json#/$defs/classification` for the rationale.

## How content loads
On load the site reads `content/manifest.json`, then fetches each file it lists and assembles the site. The manifest is an **index of which files exist** — it is generated by a script, never hand-edited.

```
content/
  site.json              site title/tagline (singleton)
  products.json          EO product taxonomy (singleton)
  products.index.json    precomputed ancestor/descendant closures (generated)
  market-segments.json   segment vocabulary + themed segment pages (singleton)
  needs/        need.*.json     one file per need/requirement
  usecases/     uc-*.json       one file per use case
  casestudies/  cs-*.json       one file per case study
  tools/        tool-*.json     one file per tool/demonstrator (case-study schema, no schema of its own)
  news/         *.json          one file per news item (own schema, kept outside the graph)
  manifest.json           GENERATED by generate_manifest.py
  _templates/             copy these to make new items (ignored by the indexer)
schemas/
  common.schema.json            shared $defs: ids, relations, classification, newsType, sources, history…
  need-requirement.schema.json
  use-case.schema.json
  case-study.schema.json        also used as-is for tools — see its description field
  news.schema.json
  vocab/product.schema.json
  vocab/market-segment.schema.json
  validate.py                    schema + referential-integrity check (needs pip install jsonschema referencing)
```

## Adding / editing content
1. Create or edit a file under `content/needs/`, `content/usecases/`, `content/casestudies/`, `content/tools/`, or `content/news/` (copy from `content/_templates/` if present). Filenames can be anything ending in `.json`; the `id` inside is what counts. Files starting with `_` are ignored.
2. To link a use case to a need, or a case study/tool to a use case, add a `relations` entry — don't add an id to some other array.
3. Run the indexer from the repo root:
   ```bash
   python generate_manifest.py
   ```
4. Commit. The item appears on the site.

To remove an item, delete its file and rerun the script. `products.json` and `market-segments.json` are single files — edit them directly and rerun the script.

The indexer **refuses to write the manifest** (exit code 1) on invalid JSON, a missing required field, or a duplicate `id`, and prints non-blocking warnings for a relation that doesn't resolve to the right kind, or a `classification` id not found in the product/segment vocabulary (the news sentinel `"all"` is exempted from that check).

For a stricter check (full JSON Schema validation, not just the indexer's lighter checks), run:
```bash
pip install jsonschema referencing
python schemas/validate.py
```

### Optional: authoring panel
`admin.html` is a convenience for editing. It loads the live content and lets you download each item as its own `<id>.json` file (plus the singletons), telling you exactly which folder to drop it in. Linking a use case to a need or a case study to a use case is done via dropdowns that write the correct `relations` entry — you don't edit the array directly. After placing files, run `generate_manifest.py`. The panel never touches the engine or the manifest. (The panel currently exposes needs/usecases/casestudies/news/segments; tools aren't in the panel yet — add/edit `content/tools/*.json` directly.)

## Pages (engine)
| File | Purpose |
|------|---------|
| `index.html` | Landing — Copernicus overview + segment entry points |
| `segments.html` | Segments overview; per-segment page (challenges + needs → use cases → case studies, role of Copernicus). Needs are resolved live from `content/needs/` filtered by `classification.market_segments`, not stored inline. |
| `needs.html` / `need.html` | User needs list (box cards, segment filter + need/requirement kind filter) / detail (statement, priority/confidence, use cases that address it via the resolved graph) |
| `usecases.html` / `usecase.html` | Use case list (maturity-marked boxes, segment filter) / detail (maturity at top, role of Copernicus from `classification.site`, related case studies via the resolved graph) |
| `casestudies.html` / `casestudy.html` | Case study list / detail (maturity badge, EU-project flag from `classification.site.eu_project`) |
| `products.html` | EO product taxonomy browsable by family, with sub-products nested under their parent |
| `tools.html` / `tool.html` | Tool/demonstrator list (segment filter) / detail (narrative split into What it does / Why it matters / Status, launch link from `classification.site.project_link` when present, otherwise a disabled "no demo yet" state) |
| `news.html` | News feed with segment filter, reading `classification.market_segments` like the rest of the content |
| `about.html` | Purpose, use-case vs case-study distinction, maturity key, EUSPA's role |
| `admin.html` | Content authoring tool (per-file export) |
| `app.js` / `style.css` | Shared engine: nav/footer/filters/loader/graph-resolution, styles |
| `generate_manifest.py` | Indexes `content/` into `content/manifest.json` |
| `trace.html` / `trace.js` | Standalone TRACE-style force-directed graph inspector (needs/use cases/case studies, drag/pan/zoom, relation+segment filters, detail panel) |
| `graph-inspector.html` | Standalone dark-mode graph inspector with item-type and relation-type toggles |

## Deploying to GitHub Pages
1. Commit everything (including `content/manifest.json` and `content/products.index.json`).
2. **Settings → Pages → Deploy from a branch → `main` / `root`**.
3. Live at `https://USER.github.io/REPO/`.

Whenever you change content, run `python generate_manifest.py` before committing so the manifest matches the files. If you edited `content/products.json`, also run `python product_taxonomy.py --write` to refresh `products.index.json`.

## Local preview
The site fetches JSON, so it needs a server (not `file://`):
```bash
python -m http.server 8000   # open http://localhost:8000
```

## Maturity levels
Use cases, case studies, and tools carry one of **Concept**, **Pilot**, **Demonstrated**, **Operational**, **Scaled**.

## Real EU projects
Case studies with `classification.site.eu_project: true` reflect documented EUSPA / Horizon Europe projects — **SPATRA** (rail temperature/buckling), **BUILDSPACE** (urban digital twins), **Space4Cities** (urban public space). Others are illustrative; their figures are indicative. The current tools are all concept-stage placeholders (no live demo yet); add `classification.site.project_link` once a real instance exists and the tool page switches to a "live demo" state automatically.

## News categories
`classification.market_segments` aside, news carries a `type` from `common.schema.json#/$defs/newsType`: `grant_call`, `cassini`, `procurement`, `product_update`, `project`, `policy`, `event`, `other`. Add new values there (not in `news.schema.json` directly) if a recurring category emerges, to keep the enum in one place.
